# Harness — Agent Team & Skill Architect

根据领域/项目配置相应的 Harness，定义各智能体的角色，并创建智能体所使用技能的元技能。

**核心原则:**
1. 创建智能体定义(`.claude/agents/`)和技能(`.claude/skills/`)。
2. **将智能体团队作为基本执行模式。**
3. **在 CLAUDE.md 中注册 Harness 指针。** — 仅记录最小化的指针（触发规则 + 变更履历），以便在新会话中触发编排器技能。
4. **Harness 不是固定产物，而是持续进化的系统。** — 每次执行后反映反馈，持续更新智能体、技能和 CLAUDE.md。

## 工作流

### Phase 0: 现状审计

Harness 技能被触发后，首先确认现有 Harness 的现状。

1. 读取 `项目/.claude/agents/`、`项目/.claude/skills/`、`项目/CLAUDE.md`
2. 根据现状分叉执行模式:
   - **新建搭建**: 智能体/技能目录不存在或为空 → 从 Phase 1 开始完整执行
   - **现有扩展**: 已有 Harness 且请求添加新智能体/技能 → 根据下方 Phase 选择矩阵仅执行所需 Phase
   - **运维/维护**: 对现有 Harness 的审计、修改、同步请求 → 进入 Phase 7-5 运维/维护工作流

   **现有扩展时的 Phase 选择矩阵:**
   | 变更类型 | Phase 1 | Phase 2 | Phase 3 | Phase 4 | Phase 5 | Phase 6 |
   |----------|---------|---------|---------|---------|---------|---------|
   | 添加智能体 | 跳过 (利用 Phase 0 结果) | 仅分配决策 | 必需 (含 3-0) | 需要专用技能时 (含 4-0) | 修改编排器 | 必需 |
   | 添加/修改技能 | 跳过 | 跳过 | 跳过 | 必需 (含 4-0) | 连接变更时 | 必需 |
   | 架构变更 | 跳过 | 必需 | 仅受影响的智能体 (含 3-0) | 仅受影响的技能 (含 4-0) | 必需 | 必需 |
3. 将现有智能体/技能列表与 CLAUDE.md 记录进行对照，检测不一致（drift）
4. 向用户汇总报告审计结果，确认执行计划

### Phase 1: 领域分析
1. 从用户请求中识别领域/项目
2. 识别核心任务类型（创建、验证、编辑、分析等）
3. 基于 Phase 0 审计结果分析现有智能体/技能的冲突/重复
4. 探索项目代码库 — 识别技术栈、数据模型、主要模块
5. **用户熟练度检测** — 通过对话的上下文线索（使用术语、提问水平）判断技术水平，调整后续沟通语调。对编码经验较少的用户，避免不加解释地使用 "assertion"、"JSON schema" 等术语。

### Phase 2: 团队架构设计

#### 2-1. 执行模式选择

**智能体团队是最优先的默认值。** 当 2 个以上智能体协作时，必须优先考虑智能体团队。团队成员之间通过直接通信（SendMessage）和共享任务列表（TaskCreate）进行自主协调，发现共享、冲突讨论和遗漏补充能提升产出质量。

| 模式 | 何时使用 | 特性 |
|------|----------|------|
| **智能体团队** (默认) | 2 人以上协作，需要实时协调、反馈交换，中间产出物交叉引用 | 通过 `TeamCreate` + `SendMessage` + `TaskCreate` 自主协调 |
| **子智能体** (替代方案) | 单一智能体任务，仅需向主智能体返回结果，团队通信开销过大时 | 直接调用 `Agent` 工具，使用 `run_in_background` 并行 |
| **混合模式** | 各 Phase 特性不同时 — 例如：并行采集（子） → 共识整合（团队） | 按 Phase 混合组成团队/子智能体 |

**决策顺序:**
1. 首先评估是否可以用智能体团队设计 — 2 人以上则使用默认值
2. 仅当团队通信在结构上不必要（仅需结果传递），且团队开销大于收益时，选择子智能体
3. 各 Phase 特性明显不同时考虑混合模式 — 在编排器中明确各 Phase 的执行模式

> 详细对比表和模式决策树参见 `references/agent-design-patterns.md` 的"执行模式"部分。

#### 2-2. 架构模式选择

1. 将任务分解为专业领域
2. 确定智能体团队结构（架构模式参见 `references/agent-design-patterns.md`）
   - **流水线**: 顺序依赖任务
   - **扇出/扇入**: 并行独立任务
   - **专家池**: 按场景选择调用
   - **生成-验证**: 生成后质量检验
   - **监督者**: 中央智能体管理状态及动态分配
   - **层级委派**: 上层智能体向下层递归委派

#### 2-3. 智能体分离标准

从专业性、并行性、上下文、复用性四个维度判断。详细标准表参见 `references/agent-design-patterns.md` 的"智能体分离标准"部分。与现有智能体的重复和复用审查在 Phase 3-0 中处理。

### Phase 3: 智能体定义生成

#### 3-0. 现有智能体重复审查

在创建新智能体之前，检查 `项目/.claude/agents/` 中现有智能体是否存在重复。反复搭建 Harness 时，角色重叠的智能体容易以不同名称累积。

> 重复分类标准和复用设计参见 `references/agent-design-patterns.md` 的"智能体复用设计"部分。

**所有智能体必须通过 `项目/.claude/agents/{name}.md` 文件定义。** 禁止不创建智能体定义文件而直接在 Agent 工具的 prompt 中写入角色。原因：
- 智能体定义必须以文件形式存在才能在下次会话中复用
- 必须明确团队通信协议才能保证智能体间协作质量
- Harness 的核心价值在于智能体（谁来做）和技能（怎么做）的分离

即使使用内置类型（`general-purpose`、`Explore`、`Plan`），也要创建智能体定义文件。内置类型通过 Agent 工具的 `subagent_type` 参数指定，智能体定义文件中包含角色、原则和协议。

**模型设置:** 所有智能体使用 `model: "opus"`。调用 Agent 工具时必须明确指定 `model: "opus"` 参数。Harness 的质量直接取决于智能体的推理能力，opus 确保最高质量。

**团队重组:** 智能体团队每次会话只能激活一个团队，但可以在 Phase 之间解散团队并组建新团队。当需要像流水线模式那样各 Phase 使用不同的专家组合时，将前一个团队的产出物保存到文件后清理团队，再创建新团队。

将各智能体定义在 `项目/.claude/agents/{name}.md` 中。必需部分：核心角色、工作原则、输入/输出协议、错误处理、协作。在智能体团队模式下，添加 `## 团队通信协议` 部分，明确消息收发对象和任务请求范围。

> 定义模板和实际文件全文参见 `references/agent-design-patterns.md` 的"智能体定义结构" + `references/team-examples.md`。

**包含 QA 智能体时的必需事项：**
- QA 智能体使用 `general-purpose` 类型（`Explore` 是只读的，无法运行验证脚本）
- QA 的核心不是"确认存在"而是**"边界交叉比较"** — 同时读取 API 响应和前端 hook 并比较 shape
- QA 不是全部完成后执行 1 次，而是**各模块完成后渐进执行**（incremental QA）
- 详细指南参见 `references/qa-agent-guide.md`

### Phase 4: 技能生成

将各智能体使用的技能创建在 `项目/.claude/skills/{name}/SKILL.md` 中。详细编写指南参见 `references/skill-writing-guide.md`。

#### 4-0. 现有技能重复审查

在创建新技能之前，检查 `项目/.claude/skills/` 中现有技能是否存在重复。反复搭建 Harness 时，功能重叠的技能容易以不同名称累积。

> 重复分类标准和泛化模式参见 `references/skill-writing-guide.md` 的"技能复用设计"部分。

#### 4-1. 技能结构

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML frontmatter (name, description 必需)
│   └── Markdown 正文
└── Bundled Resources (可选)
    ├── scripts/    - 重复/确定性任务的执行代码
    ├── references/ - 条件加载的参考文档
    └── assets/     - 输出所用的文件（模板、图片等）
```

#### 4-2. Description 编写 — 积极触发诱导

description 是技能唯一的触发机制。Claude 倾向于保守判断触发，因此 description 要编写得**积极（"pushy"）**。

**反面示例:** `"处理 PDF 文档的技能"`
**正面示例:** `"执行 PDF 文件读取、文本/表格提取、合并、拆分、旋转、水印、加密、OCR 等所有 PDF 操作。提及 .pdf 文件或请求 PDF 产出物时务必使用此技能。"`

核心：同时描述技能的功能 + 具体触发场景，并与相似但不应触发的情况区分开来。

#### 4-3. 正文编写原则

| 原则 | 说明 |
|------|------|
| **说明原因** | 不使用 "ALWAYS/NEVER" 等强制性指令，而是传达为什么要这样做。LLM 理解原因后能在边缘情况下做出正确判断。 |
| **保持精简** | 上下文窗口是公共资源。SKILL.md 正文目标在 500 行以内，不增加重量的内容删除或移至 references/。 |
| **泛化** | 与其编写仅适用于特定示例的狭窄规则，不如说明原理以应对各种输入。禁止过拟合。 |
| **重复代码打包** | 如果在测试执行中发现智能体们共同编写的脚本，预先打包到 `scripts/` 中。 |
| **使用祈使语气** | 使用"做..."、"请..."形式的祈使/指令语调。 |

#### 4-4. Progressive Disclosure (渐进式信息展示)

技能通过 3 级加载系统管理上下文：

| 级别 | 加载时机 | 大小目标 |
|------|----------|----------|
| **Metadata** (name + description) | 始终存在于上下文中 | 约 100 词 |
| **SKILL.md 正文** | 技能触发时 | <500 行 |
| **references/** | 仅在需要时 | 无限制（脚本无需加载即可执行） |

**大小管理规则：**
- SKILL.md 接近 500 行时，将详细内容分离到 references/，在正文中留下"何时读取此文件"的指针
- 300 行以上的 reference 文件在顶部包含**目录（ToC）**
- 如有领域/框架的变体，在 references/ 下按领域分离，仅加载相关文件

```
cloud-deploy/
├── SKILL.md (工作流 + 选择指南)
└── references/
    ├── aws.md    ← 仅在选择 AWS 时加载
    ├── gcp.md
    └── azure.md
```

#### 4-5. 技能-智能体连接原则

- 1 个智能体 ↔ 1~N 个技能（1:1 或 1:多）
- 多个智能体可以共享同一技能
- 技能承载"怎么做"，智能体承载"谁来做"

> 详细编写模式、示例和数据 schema 标准参见 `references/skill-writing-guide.md`。

### Phase 5: 集成与编排

编排器是技能的特殊形式，将各个智能体和技能编织成一个工作流来协调整个团队。Phase 4 创建的各个技能定义"各智能体做什么、怎么做"，而编排器定义"谁在何时以什么顺序协作"。具体模板参见 `references/orchestrator-template.md`。

**现有扩展时编排器修改：** 当不是新建而是现有扩展时，不创建新编排器而是修改现有编排器。添加智能体时在团队构成、任务分配、数据流中反映新智能体，并在 description 中添加与新智能体相关的触发关键词。

根据 Phase 2-1 中选择的执行模式，编排器模式有所不同：

#### 5-0. 编排器模式（按模式）

**智能体团队模式（默认）：**
编排器通过 `TeamCreate` 组建团队，通过 `TaskCreate` 分配任务。团队成员通过 `SendMessage` 直接通信进行自主协调。领导者（编排器）监控进展并汇总结果。

```
[编排器/领导者]
    ├── TeamCreate(team_name, members)
    ├── TaskCreate(tasks with dependencies)
    ├── 团队成员自主协调 (SendMessage)
    ├── 收集并汇总结果
    └── 清理团队
```

**子智能体模式（替代方案）：**
编排器通过 `Agent` 工具直接调用子智能体。并行执行使用 `run_in_background: true`，结果仅返回给主智能体。当团队通信不必要且想减少开销时使用。

```
[编排器]
    ├── Agent(agent-1, run_in_background=true)
    ├── Agent(agent-2, run_in_background=true)
    ├── 等待并收集结果
    └── 生成整合产出物
```

**混合模式：**
各 Phase 混合不同模式。常用组合：
- **并行采集（子） → 共识整合（团队）**: Phase 2 用子智能体并行采集独立资料 → Phase 3 组建团队进行讨论和共识整合
- **团队生成（团队） → 验证（子）**: Phase 2 团队生成初稿 → Phase 3 单一子智能体独立验证
- **Phase 间团队重组**: 各 Phase 执行 `TeamDelete` 后新 `TeamCreate`，中间插入子智能体调用

选择混合模式时，在编排器各 Phase 部分顶部明确该 Phase 的执行模式（例如：`**执行模式:** 智能体团队`）。

#### 5-1. 数据传递协议

在编排器中明确智能体间的数据传递方式：

| 策略 | 方式 | 适用模式 | 适用场景 |
|------|------|----------|----------|
| **消息驱动** | 通过 `SendMessage` 在团队成员间直接通信 | 团队 | 实时协调、反馈交换、轻量状态传递 |
| **任务驱动** | 通过 `TaskCreate`/`TaskUpdate` 共享任务状态 | 团队 | 进展跟踪、依赖关系管理、任务自身请求 |
| **文件驱动** | 在约定路径写入和读取文件 | 团队 + 子 | 大容量数据、结构化产出物、需要审计追踪 |
| **返回值驱动** | `Agent` 工具的返回消息 | 子 | 主智能体直接收集子智能体结果 |

**推荐组合（团队模式）：** 任务驱动（协调）+ 文件驱动（产出物）+ 消息驱动（实时沟通）
**推荐组合（子模式）：** 返回值驱动（结果收集）+ 文件驱动（大容量产出物）
**混合模式：** 根据各 Phase 的执行模式应用相应组合

文件驱动传递时的规则：
- 在工作目录下创建 `_workspace/` 文件夹存储中间产出物
- 文件命名约定：`{phase}_{agent}_{artifact}.{ext}`（例如：`01_analyst_requirements.md`）
- 最终产出物仅输出到用户指定路径，中间文件（`_workspace/`）保留（用于事后验证和审计追踪）

#### 5-2. 错误处理

在编排器中包含错误处理方针。核心原则：重试 1 次后若再失败则跳过该结果继续（在报告中注明缺失），冲突数据不删除而是标注来源。

> 错误类型策略表和实现详情参见 `references/orchestrator-template.md` 的"错误处理"部分。

#### 5-3. 团队规模指南

| 任务规模 | 推荐团队成员数 | 每人任务数 |
|----------|--------------|-----------|
| 小规模（5~10 个任务） | 2~3 人 | 3~5 个 |
| 中规模（10~20 个任务） | 3~5 人 | 4~6 个 |
| 大规模（20 个以上任务） | 5~7 人 | 4~5 个 |

> 团队成员越多协调开销越大。3 个专注的成员优于 5 个分散的成员。

#### 5-4. CLAUDE.md Harness 指针注册

Harness 配置完成后，在项目的 `CLAUDE.md` 中注册最小化的指针。CLAUDE.md 在每次新会话时加载，因此只需记录 Harness 存在和触发规则，编排器技能会处理其余部分。

**CLAUDE.md 模板：**

````markdown
## Harness: {领域名}

**目标:** {Harness 核心目标一行}

**触发:** 当请求 {领域} 相关工作时使用 `{orchestrator-skill-name}` 技能。简单问题可直接回答。

**变更履历:**
| 日期 | 变更内容 | 对象 | 原因 |
|------|----------|------|------|
| {YYYY-MM-DD} | 初始配置 | 全部 | - |
````

**CLAUDE.md 中不放入的内容：** 智能体列表、技能列表、目录结构、执行规则详情。原因：智能体/技能列表由编排器技能和 `.claude/agents/`、`.claude/skills/` 管理，放入会重复。目录结构可直接从文件系统确认。CLAUDE.md 仅包含**指针（触发规则）+ 变更履历**。

#### 5-5. 后续任务支持

编排器不仅要处理初始执行，还要处理后续任务。请确保以下三点：

**1. 编排器 description 中包含后续关键词：**
仅靠初始创建关键词无法触发后续请求。description 中必须包含的后续表达：
- "重新执行"、"再执行"、"更新"、"修改"、"补充"
- "仅 {领域} 的 {子任务}"
- "基于之前的结果"、"改进结果"

**2. 编排器 Phase 1 中添加上下文确认步骤：**
工作流开始时确认现有产出物是否存在，决定执行模式：
- `_workspace/` 存在 + 用户请求部分修改 → **部分重新执行**（仅重新调用相关智能体）
- `_workspace/` 存在 + 用户提供新输入 → **新执行**（将现有 _workspace 移至 `_workspace_prev/`）
- `_workspace/` 不存在 → **初始执行**

**3. 智能体定义中包含重新调用指南：**
在各智能体 `.md` 文件中明确"存在之前产出物时的行为"：
- 如果之前的结果文件存在，读取并反映改进点
- 如果给出了用户反馈，仅修改相应部分

> 编排器模板的"Phase 0: 上下文确认"部分参见：`references/orchestrator-template.md`

### Phase 6: 验证与测试

验证生成的 Harness。详细测试方法论参见 `references/skill-testing-guide.md`。

#### 6-1. 结构验证

- 确认所有智能体文件位于正确位置
- 验证技能的 frontmatter（name, description）
- 确认智能体间引用的一致性
- 确认未生成 commands

#### 6-2. 按执行模式验证

- **智能体团队**: 确认团队成员间通信路径、任务依赖、团队规模适当性
- **子智能体**: 确认各智能体的输入输出连接、`run_in_background` 设置、返回值收集逻辑
- **混合模式**: 确认各 Phase 的执行模式是否在编排器中明确，Phase 边界处数据传递是否中断（团队 → 子转换时，团队的产出物是否连接到子的输入）

#### 6-3. 技能执行测试

对生成的每个技能执行实际执行测试：

1. **编写测试提示词** — 为每个技能编写 2~3 个现实的测试提示词。使用实际用户可能输入的具体自然的语句。

2. **With-skill vs Without-skill 对比执行** — 尽可能并行执行有技能和无技能的执行来确认技能的附加价值。各生成 2 个智能体：
   - **With-skill**: 读取技能后执行任务
   - **Without-skill (baseline)**: 使用相同提示词但不使用技能执行

3. **结果评估** — 定性（用户评审）+ 定量（基于 assertion）评估产出物的质量。当产出物可以客观验证时（文件创建、数据提取等）定义 assertion，主观的情况下（文风、设计）依赖用户反馈。

4. **迭代改进循环** — 测试结果发现问题时：
   - **泛化**反馈后修改技能（禁止仅适用于特定示例的狭窄修改）
   - 修改后重新测试
   - 重复直到用户满意或不再有显著改进

5. **重复模式打包** — 测试执行中发现智能体们共同编写的代码（例如所有测试都生成相同的 helper 脚本）时，预先将该代码打包到 `scripts/` 中。

#### 6-4. 触发验证

验证各技能的 description 是否正确触发：

1. **Should-trigger 查询**（8~10 个）— 应触发技能的各种表达（正式/随意、明确/隐含）
2. **Should-NOT-trigger 查询**（8~10 个）— 关键词相似但适合其他工具/技能而非此技能的"near-miss"查询

**near-miss 编写要点：** 像"编写斐波那契函数"这样明显无关的查询没有测试价值。像"把这个 Excel 文件的图表提取为 PNG"（xlsx 技能 vs 图片转换）这样**边界模糊的查询**才是好的测试用例。

与现有技能的触发冲突也在此步骤确认。

#### 6-5. 空运行测试

- 审查编排器技能的 Phase 顺序是否逻辑合理
- 确认数据传递路径没有空白段（dead link）
- 确认所有智能体的输入与前一个 Phase 的输出匹配
- 确认各错误场景的回退路径可执行

#### 6-6. 编写测试场景

- 在编排器技能中添加 `## 测试场景` 部分
- 描述至少 1 个正常流程 + 1 个错误流程

### Phase 7: Harness 进化

Harness 不是一次创建后就结束的静态产出物。它是根据用户反馈持续进化的系统。

#### 7-1. 执行后反馈收集

每次 Harness 执行完成后，向用户征求反馈：
- "结果中有需要改进的地方吗？"
- "智能体团队构成或工作流中有想改变的吗？"

如果没有反馈则跳过。不强求，但务必提供机会。

#### 7-2. 反馈反映路径

根据反馈类型，修改对象不同：

| 反馈类型 | 修改对象 | 示例 |
|-----------|----------|------|
| 产出物质量 | 对应智能体的技能 | "分析太表面" → 在技能中添加深度标准 |
| 智能体角色 | 智能体定义 `.md` | "也需要安全审查" → 添加新智能体 |
| 工作流顺序 | 编排器技能 | "应该先验证" → 更改 Phase 顺序 |
| 团队构成 | 编排器 + 智能体 | "这两个可以合并" → 合并智能体 |
| 触发遗漏 | 技能 description | "用这个表达不行" → 扩展 description |

#### 7-3. 变更履历

所有变更记录到 CLAUDE.md 的**变更履历**表中（与 Phase 5-4 模板的"变更履历"部分同一张表）：

```markdown
**变更履历:**
| 日期 | 变更内容 | 对象 | 原因 |
|------|----------|------|------|
| 2026-04-05 | 初始配置 | 全部 | - |
| 2026-04-07 | 添加 QA 智能体 | agents/qa.md | 产出物质量验证不足的反馈 |
| 2026-04-10 | 添加语调指南 | skills/content-creator | "太生硬"的反馈 |
```

通过此履历追踪 Harness 的进化方向，防止回归（regression）。

#### 7-4. 进化触发

不仅限于用户明确说"修改一下 Harness"的时候，以下情况也应提议进化：
- 相同类型的反馈重复 2 次以上时
- 发现智能体反复失败的模式时
- 观察到用户绕过编排器手动操作时

#### 7-5. 运维/维护工作流

系统性地执行现有 Harness 的检查、修改和同步。当 Phase 0 中进入"运维/维护"分支时，遵循此工作流。

**Step 1: 现状审计**
- 将 `.claude/agents/` 文件列表与编排器技能的智能体构成进行比较 → 生成不一致列表
- 将 `.claude/skills/` 目录列表与编排器技能的技能构成进行比较 → 生成不一致列表
- 向用户报告审计结果

**Step 2: 渐进式添加/修改**
- 根据用户请求执行智能体添加/修改/删除，技能添加/修改/删除
- 变更一次一个，每次变更后立即执行 Step 3（同步）

**Step 3: 更新 CLAUDE.md 变更履历**
- 在变更履历表中记录日期、变更内容、对象、原因

**Step 4: 变更验证**
- 修改后的智能体/技能的结构验证（以 Phase 6-1 为标准）
- 修改范围影响触发时执行触发验证（以 Phase 6-4 为标准）
- 大规模变更（架构变更、添加/删除 3 个以上智能体）时执行到 Phase 6-3（执行测试）、6-5（空运行）
- 最终确认 CLAUDE.md 与实际文件的一致性
