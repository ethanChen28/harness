## 1. 测试框架概览

技能质量验证是**定性评估**与**定量评估**的组合。

| 评估类型 | 方法 | 适用技能 |
|----------|------|----------|
| **定性** | 用户直接审阅产出物 | 文体、设计、创作物等主观质量 |
| **定量** | assertion 基于的自动评分 | 文件生成、数据提取、代码生成等可客观验证的 |

核心循环: **编写 → 执行测试 → 评估 → 改进 → 重新测试**

## 2. 测试提示词编写方法

### 原则

测试提示词应当是**实际用户可能输入的具体、自然的语句**。抽象或人工的提示词测试价值低。

### 反面示例

```
"处理 PDF"
"提取数据"
"生成图表"
```

### 正面示例

```
"从下载文件夹的 'Q4_营收_最终_v2.xlsx' 中，使用 C 列(营收)和 D 列(成本)
添加利润率(%)列。然后按利润率降序排列。"
```

```
"从这个 PDF 中提取第3页的表格并转换为 CSV。表头有两行，
第一行是类别，第二行才是实际的列名。"
```

### 提示词多样性

- 混合**正式/随意**语气
- 混合**显式/隐式**意图 (直接说出文件格式的情况 vs 需要从上下文推断的情况)
- 混合**简单/复杂**任务
- 部分包含缩写、拼写错误、随意表达

### 覆盖范围

从2~3个提示词开始，设计覆盖以下内容:
- 核心使用场景 1个
- 边界情况 1个
- (可选) 复合任务 1个

## 3. 执行测试: With-skill vs Baseline

### 3-1. 对比执行结构

对每个测试提示词**同时**生成两个子智能体:

**With-skill 执行:**
```
提示词: "{测试提示词}"
技能路径: {技能路径}
输出路径: _workspace/iteration-N/eval-{id}/with_skill/outputs/
```

**Baseline 执行:**
```
提示词: "{测试提示词}"  (相同)
技能: 无
输出路径: _workspace/iteration-N/eval-{id}/without_skill/outputs/
```

### 3-2. Baseline 选择

| 情况 | Baseline |
|------|----------|
| 创建新技能 | 不使用技能执行相同提示词 |
| 改进已有技能 | 修改前的技能版本 (保留快照) |

### 3-3. 时间数据捕获

在子智能体完成通知中**立即**保存 `total_tokens` 和 `duration_ms`。这些数据仅在通知时刻可访问，之后无法恢复。

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

## 4. 定量评估: Assertion 基于的评分

### 4-1. Assertion 编写

当产出物可客观验证时，定义用于自动评分的 assertion。

**好的 assertion:**
- 可客观判断真/假
- 使用描述性名称，仅看结果就能清楚检查什么
- 验证技能的核心价值

**不好的 assertion:**
- 无论有无技能都始终通过的 (如 "输出存在")
- 需要主观判断的 (如 "写得好")

### 4-2. 可编程的验证

如果 assertion 可通过代码验证，则编写脚本。比目视确认更快更可靠，且每次迭代可复用。

### 4-3. 注意 Non-discriminating assertion

"两种配置都100%通过" 的 assertion 无法衡量技能的差异化价值。发现这类 assertion 时应移除或替换为更具挑战性的 assertion。

### 4-4. 评分结果 Schema

```json
{
  "expectations": [
    {
      "text": "利润率列已添加",
      "passed": true,
      "evidence": "确认 E 列有 'profit_margin_pct' 列"
    },
    {
      "text": "按利润率降序排列",
      "passed": false,
      "evidence": "未排序，保持原始顺序"
    }
  ],
  "summary": {
    "passed": 1,
    "failed": 1,
    "total": 2,
    "pass_rate": 0.50
  }
}
```

## 5. 专家智能体的运用

在测试/评估过程中利用专家角色的智能体可提升质量。

### 5-1. Grader (评分者)

执行基于 assertion 的评分，从产出物中提取可验证的声明 (claim) 并进行交叉验证。

**角色:**
- 每个 assertion 的通过/失败判定 + 提供证据
- 从产出物中提取事实性声明并验证
- 对评估本身质量的反馈 (当 assertion 太简单或模糊时提出建议)

### 5-2. Comparator (盲评比较者)

将两个产出物匿名化为 A/B，在不知道哪个使用了技能的情况下判定质量。

**使用时机:** 想严格确认 "新版本是否真的更好？" 时。一般迭代改进中可省略。

**判定标准:**
- 内容: 准确性，完成度
- 结构: 组织性，格式化，易用性
- 综合评分

### 5-3. Analyzer (分析者)

从基准数据中分析统计模式:
- Non-discriminating assertion (两种配置都通过 → 无区分力)
- 高方差评估 (结果每次执行变化很大 → 不稳定)
- 时间/Token 权衡 (技能提高了质量但也增加了成本)

## 6. 迭代改进循环

### 6-1. 收集反馈

向用户展示产出物并获取反馈。空反馈解读为 "无异常"。

### 6-2. 改进原则

1. **泛化反馈** — 仅针对测试示例的狭窄修改是过拟合。应在原理层面修改。
2. **移除不产生价值的部分** — 阅读转录记录，如果技能让智能体执行了非生产性工作，则删除该部分。
3. **解释 Why** — 即使用户的反馈很简洁，也要理解为什么重要，并将这种理解反映到技能中。
4. **打包重复工作** — 如果每次测试执行都生成相同的辅助脚本，则预先包含在 `scripts/` 中。

### 6-3. 迭代步骤

```
1. 修改技能
2. 在新的 iteration-N+1/ 目录中重新执行所有测试用例
3. 向用户展示结果 (与之前迭代对比)
4. 收集反馈
5. 再次修改 → 循环
```

**终止条件:**
- 用户满意
- 所有反馈为空 (所有产出物无异常)
- 不再有有意义的改进

### 6-4. 草稿 → 重新审视模式

修改技能时，写完草稿后**以新的视角重新阅读并改进**。不要试图一次写完美，而是经过草稿-审视循环。

## 7. Description 触发验证

### 7-1. 编写触发评估查询

编写20个评估查询 — should-trigger 10个 + should-NOT-trigger 10个。

**查询质量标准:**
- 实际用户可能输入的具体、自然的语句
- 包含文件路径、个人上下文、列名、公司名等具体细节
- 长度、语气、格式多样化混合
- 聚焦**边界情况 (edge case)** 而非明确正确答案

**Should-trigger 查询 (8~10个):**
- 相同意图的不同表达 (正式/随意)
- 未明确说出技能/文件类型但显然需要的情况
- 非主流使用场景
- 与其他技能竞争但本技能应胜出的情况

**Should-NOT-trigger 查询 (8~10个):**
- **Near-miss 是核心** — 关键词相似但适合其他工具/技能的查询
- 明显无关的查询 ("写斐波那契函数") 没有测试价值
- 相邻领域、模糊表达、关键词重叠但上下文不同的情况

### 7-2. 已有技能冲突验证

确认新技能的 description 不与已有技能的触发区域重叠:

1. 收集已有技能列表的 description
2. 确认新技能的 should-trigger 查询不会错误触发已有技能
3. 发现冲突时更清晰地描述 description 的边界条件

### 7-3. 自动优化 (可选高级功能)

需要优化 description 时:

1. 将20个评估查询分为 Train(60%) / Test(40%)
2. 用当前 description 测量触发准确度
3. 分析失败案例，生成改进的 description
4. 以 Test set 为标准选择最佳 description (而非 Train set — 防止过拟合)
5. 最多重复5次

> 此过程使用 `claude -p` 的自动化脚本执行。Token 成本较高，因此应在技能充分稳定后的最终阶段执行。

## 8. 工作区结构

系统化管理测试/评估结果的目录结构:

```
{skill-name}-workspace/
├── iteration-1/
│   ├── eval-descriptive-name-1/
│   │   ├── eval_metadata.json
│   │   ├── with_skill/
│   │   │   ├── outputs/
│   │   │   ├── timing.json
│   │   │   └── grading.json
│   │   └── without_skill/
│   │       ├── outputs/
│   │       ├── timing.json
│   │       └── grading.json
│   ├── eval-descriptive-name-2/
│   │   └── ...
│   └── benchmark.json
├── iteration-2/
│   └── ...
└── evals/
    └── evals.json
```

**规则:**
- 评估目录使用**描述性名称**而非数字 (如 `eval-multi-page-table-extraction`)
- 每次迭代保存在独立目录中 (禁止覆盖之前迭代)
- `_workspace/` 不删除 — 用于事后验证及审计追踪
