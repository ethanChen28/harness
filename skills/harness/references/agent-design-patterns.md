# 智能体团队设计模式

## 执行模式: 智能体团队 vs 子智能体

理解两种执行模式的核心差异，选择合适的模式。

### 智能体团队 (Agent Teams) — 默认模式

团队领导者通过 `TeamCreate` 组建团队，团队成员作为独立的 Claude Code 实例运行。团队成员通过 `SendMessage` 直接通信，通过共享任务列表 (`TaskCreate`/`TaskUpdate`) 进行自主协调。

```
[领导者] ←→ [成员A] ←→ [成员B]
  ↕          ↕          ↕
  └──── 共享任务列表 ────┘
```

**核心工具:**
- `TeamCreate`: 创建团队 + 生成团队成员
- `SendMessage({to: name})`: 向特定成员发送消息
- `SendMessage({to: "all"})`: 广播 (成本高，少用)
- `TaskCreate`/`TaskUpdate`: 管理共享任务列表

**特征:**
- 成员之间可直接对话、质疑、验证
- 无需经过领导者即可在成员之间交换信息
- 通过共享任务列表进行自主协调 (可自行认领任务)
- 成员进入空闲状态时自动通知领导者
- 可通过计划审批模式在危险操作前进行审查

**限制:**
- 每个会话只能**激活**一个团队 (但可在 Phase 之间解散团队并组建新团队)
- 不允许嵌套团队 (成员不能创建自己的团队)
- 领导者固定 (不可转让)
- Token 成本较高

**团队重组模式:**
如果不同 Phase 需要不同的专家组合，按以下顺序进行: 将前一个团队的产出物保存为文件 → 清理团队 → 创建新团队。前一个团队的产出物保存在 `_workspace/` 中，新团队可通过 Read 访问。

### 子智能体 (Sub-agents) — 轻量模式

主智能体通过 `Agent` 工具创建子智能体。子智能体将任务结果仅返回给主智能体，彼此之间不通信。

```
[主智能体] → [子A] → 返回结果
           → [子B] → 返回结果
           → [子C] → 返回结果
```

**核心工具:**
- `Agent(prompt, subagent_type, run_in_background)`: 创建子智能体

**特征:**
- 轻量且快速
- 结果以摘要形式返回主上下文
- Token 效率高

**限制:**
- 子智能体之间无法通信
- 主智能体负责所有协调
- 无法进行实时协作/质疑

### 模式选择决策树

```
需要2个以上智能体吗？
├── 是 → 智能体之间需要通信吗？
│         ├── 是 → 智能体团队 (默认)
│         │         交叉验证·发现共享·实时反馈提升质量。
│         │
│         └── 否 → 子智能体也可以
│                  只需传递结果的生成-验证、专家池等。
│
└── 否 (1个) → 子智能体
              单一智能体无需组建团队。
```

> **核心原则:** 智能体团队是默认选择。选择子智能体时应自问: "成员之间的通信真的不需要吗？"

## 智能体团队架构类型

### 1. 流水线 (Pipeline)
顺序工作流。前一个智能体的输出作为后一个智能体的输入。

```
[分析] → [设计] → [实现] → [验证]
```

**适用场景:** 各步骤对前一步骤的产出物有强依赖
**示例:** 小说写作 — 世界观 → 角色 → 情节 → 撰写 → 编辑
**注意:** 瓶颈会延迟整条流水线。各步骤应尽量独立设计。
**团队模式适用性:** 顺序依赖性强，团队模式的优势有限。但流水线内有并行区间时团队模式有用。

### 2. 扇出/扇入 (Fan-out/Fan-in)
并行处理后整合结果。同时执行独立任务。

```
         ┌→ [专家A] ─┐
[分发] → ├→ [专家B] ─┼→ [整合]
         └→ [专家C] ─┘
```

**适用场景:** 需要对同一输入从不同视角/领域进行分析
**示例:** 综合研究 — 官方/媒体/社区/背景同时调查 → 整合报告
**注意:** 整合步骤的质量决定整体质量。
**团队模式适用性:** 智能体团队最自然的模式。**必须以智能体团队方式组建。** 成员之间共享发现并互相质疑，一个智能体的发现可实时修正其他智能体的调查方向，相比单独调查质量大幅提升。

### 3. 专家池 (Expert Pool)
根据情况选择调用合适的专家。

```
[路由器] → { 专家A | 专家B | 专家C }
```

**适用场景:** 根据输入类型需要不同处理
**示例:** 代码审查 — 从安全/性能/架构专家中调用相关领域
**注意:** 路由器的分类准确度是关键。
**团队模式适用性:** 子智能体更合适。只调用需要的专家，无需常驻团队。

### 4. 生成-验证 (Producer-Reviewer)
生成智能体与验证智能体成对运作。

```
[生成] → [验证] → (有问题时) → [生成] 重新执行
```

**适用场景:** 产出物质量保证重要且存在客观验证标准
**示例:** 网络漫画 — artist 生成 → reviewer 审核 → 问题面板重新生成
**注意:** 为防止无限循环，必须设置最大重试次数 (2~3次)。
**团队模式适用性:** 智能体团队有用。通过 SendMessage 在生成者↔验证者之间交换实时反馈。

### 5. 监督者 (Supervisor)
中央智能体管理工作状态，动态向下级智能体分配任务。

```
         ┌→ [工作者A]
[监督者] ─┼→ [工作者B]    ← 监督者根据状态动态分配
         └→ [工作者C]
```

**适用场景:** 工作量可变或需要在运行时决定任务分配
**示例:** 大规模代码迁移 — 监督者分析文件列表并向工作者分配批次
**与扇出的区别:** 扇出是预先固定分配任务，监督者根据进展动态调整
**注意:** 委派单位要设得足够大，避免监督者成为瓶颈。
**团队模式适用性:** 智能体团队的共享任务列表与监督者模式自然匹配。通过 TaskCreate 注册任务，成员自行认领。

### 6. 层级委派 (Hierarchical Delegation)
上级智能体递归地向下级智能体委派。将复杂问题逐步分解。

```
[总管] → [组长A] → [执行者A1]
                  → [执行者A2]
       → [组长B] → [执行者B1]
```

**适用场景:** 问题可以自然地按层级分解
**示例:** 全栈应用开发 — 总管 → 前端组长 → (UI/逻辑/测试) + 后端组长 → (API/DB/测试)
**注意:** 深度超过3层会导致延迟和上下文丢失增大。建议2层以内。
**团队模式适用性:** 智能体团队不允许嵌套 (成员不能创建团队)。第1层用团队，第2层用子智能体实现，或扁平化为单一团队。

## 复合模式

实际应用中复合模式比单一模式更常见:

| 复合模式 | 组成 | 示例 |
|----------|------|------|
| **扇出 + 生成-验证** | 并行生成后各自验证 | 多语言翻译 — 4种语言并行翻译 → 各自由母语审校者审核 |
| **流水线 + 扇出** | 顺序步骤中部分并行化 | 分析(顺序) → 实现(并行) → 集成测试(顺序) |
| **监督者 + 专家池** | 监督者动态调用专家 | 客户咨询处理 — 监督者分类咨询后分配合适的专家 |

### 复合模式中的执行模式

**基本上所有复合模式都使用智能体团队。** 成员之间的活跃沟通是结果质量的核心驱动力。

| 场景 | 推荐模式 | 原因 |
|------|----------|------|
| **研究 + 分析** | 智能体团队 | 调查者之间共享发现，对冲突信息实时讨论 |
| **设计 + 实现 + 验证** | 智能体团队 | 设计者↔实现者↔验证者之间的反馈循环 |
| **监督者 + 工作者** | 智能体团队 | 通过共享任务列表动态分配，工作者之间共享进度 |
| **生成 + 验证** | 智能体团队 | 生成者↔验证者之间实时反馈减少返工 |

> 仅在单一智能体执行完全隔离的一次性任务时才考虑混合子智能体。

## 智能体类型选择

调用智能体时通过 Agent 工具的 `subagent_type` 参数指定类型。智能体团队的成员也可以使用自定义智能体定义。

### 内置类型

| 类型 | 工具访问 | 适用用途 |
|------|----------|----------|
| `general-purpose` | 全部 (包括 WebSearch, WebFetch) | 网络调查，通用任务 |
| `Explore` | 只读 (无 Edit/Write) | 代码库探索，分析 |
| `Plan` | 只读 (无 Edit/Write) | 架构设计，计划制定 |

### 自定义类型

在 `.claude/agents/{name}.md` 中定义智能体后，可通过 `subagent_type: "{name}"` 调用。自定义智能体可访问全部工具。

### 选择标准

| 场景 | 推荐 | 原因 |
|------|------|------|
| 角色复杂且在多个会话中复用 | **自定义类型** (`.claude/agents/`) | 通过文件管理人格和工作原则 |
| 简单调查/收集且提示词足够 | **`general-purpose`** + 详细提示词 | 无需智能体文件，指令包含在提示词中 |
| 只需读取代码 (分析/审查) | **`Explore`** | 防止意外修改文件 |
| 只需设计/规划 | **`Plan`** | 专注分析，防止代码变更 |
| 需要修改文件的实现工作 | **自定义类型** | 全部工具访问 + 专门指令 |

**原则:** 所有智能体都必须通过 `.claude/agents/{name}.md` 文件定义。即使是内置类型，也应创建智能体定义文件来明确角色·原则·协议。只有作为文件存在才能在后续会话中复用，且只有明确团队通信协议才能保证协作质量。

**模型:** 所有智能体使用 `model: "opus"`。调用 Agent 工具时必须指定 `model: "opus"` 参数。

```markdown

name: agent-name
description: "1-2句话角色说明。列出触发关键词。"
---

2. 角色2

- 原则2

- 输入: [从哪里接收什么]
- 输出: [向哪里写入什么]
```

## 智能体分离标准

| 标准 | 分离 | 合并 |
|------|------|------|
| 专业性 | 领域不同则分离 | 领域重叠则合并 |
| 并行性 | 可独立执行则分离 | 顺序依赖则考虑合并 |
| 上下文 | 上下文负担大则分离 | 轻量快速则合并 |
| 复用性 | 其他团队也使用则分离 | 仅本团队使用则考虑合并 |

## 智能体复用设计

创建新智能体前，确认与现有智能体的重叠。反复构建 Harness 容易导致角色重叠的智能体以不同名称累积。

| 情况 | 措施 |
|------|------|
| 现有智能体完全包含新角色 | 禁止新建 — 复用现有智能体 |
| 现有智能体部分包含且可泛化 | 扩展现有智能体进行泛化 |
| 部分包含但有意进行领域特化 | 新建 — 作为独立智能体维护 |
| 角色范围完全不同 | 新建 |

**原则:** 一个智能体专注于一个角色时复用性最高、重复最少。如果角色有两个以上，先考虑是否可以分离。

**泛化现有智能体时:** 依赖该智能体的编排器·团队配置的行为可能发生变化。扩展前确认依赖关系，泛化后通过空运行确认现有行为得以保持。

## 技能 vs 智能体区分

| 区分 | 技能 (Skill) | 智能体 (Agent) |
|------|-------------|----------------|
| 定义 | 过程性知识 + 工具打包 | 专家人格 + 行为原则 |
| 位置 | `.claude/skills/` | `.claude/agents/` |
| 触发 | 用户请求关键词匹配 | 通过 Agent 工具显式调用 |
| 大小 | 小~大 (工作流) | 小 (角色定义) |
| 用途 | "怎么做" | "谁来做" |

技能是智能体执行任务时参考的**过程性指南**。
智能体是利用技能的**专家角色定义**。

## 技能 ↔ 智能体连接方式

智能体利用技能的3种方式:

| 方式 | 实现 | 适用场景 |
|------|------|----------|
| **Skill 工具调用** | 智能体提示词中指定 `通过 Skill 工具调用 /skill-name` | 技能是独立工作流且用户可调用的情况 |
| **提示词内联** | 智能体定义内直接包含技能内容 | 技能简短 (50行以下) 且为该智能体专用的情况 |
| **参考文档加载** | 需要时通过 `Read` 加载技能的 references/ 文件 | 技能内容大且仅条件性需要的情况 |

推荐: 复用性高用 Skill 工具，专用则内联，大容量用参考文档加载。
