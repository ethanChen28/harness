## 1. Description 编写模式

Description 是技能唯一的触发机制。Claude 从 `available_skills` 列表中仅根据 name + description 决定是否使用技能。

### 理解触发机制

Claude 倾向于不调用技能来处理可以用其基础工具轻松完成的简单任务。"读一下这个 PDF" 这样的简单请求即使 description 完美也可能不触发。任务越复杂、越多步骤、越专业，技能触发的概率越高。

### 编写原则

1. 同时描述**技能做什么** + **具体触发场景**
2. 明确区分相似但不应触发的情况的边界条件
3. 稍微 "强势" 一些 — 补偿 Claude 倾向于保守判断触发的倾向

### 正面示例

```yaml
description: "PDF 文件读取、文本/表格提取、合并、分割、旋转、水印、
  加密/解密、OCR 等所有 PDF 操作。提及 .pdf 文件或
  请求 PDF 产出物时必须使用此技能。需要转换/编辑/分析
  而非简单 '读取' PDF 时尤其有用。"
```

```yaml
description: "Excel/CSV/TSV 文件的列添加、公式计算、格式化、图表、
  数据清洗等所有电子表格操作。用户提及电子表格
  文件时 — 即使只是随意提到 ('下载文件夹的 xlsx') — 
  也请使用此技能。"
```

### 反面示例

- `"处理数据的技能"` — 太模糊，不清楚什么文件/操作
- `"PDF 相关操作"` — 没有列出具体动作，未描述触发场景

## 2. 正文编写风格

### Why-First 原则

LLM 理解原因后即使在边界情况下也能正确判断。传达上下文比强制规则更有效。

**反面示例:**
```markdown
ALWAYS use pdfplumber for table extraction. NEVER use PyPDF2 for tables.
```

**正面示例:**
```markdown
表格提取使用 pdfplumber。PyPDF2 专注于文本提取，
无法保留表格的行/列结构。pdfplumber 能识别单元格边界
并返回结构化数据。
```

### 泛化原则

当从反馈或测试结果中发现问题时，不应仅针对特定示例做狭窄修改，而应在**原理层面泛化**。

**过拟合修改:**
```markdown
如果存在 "Q4 营收" 列，则将该列转换为数字。
```

**泛化修改:**
```markdown
如果列名包含 "营收"、"金额"、"数量" 等暗示数值的关键词，
则将该列转换为数字类型。转换失败时保留原始值。
```

### 祈使语气

使用"做..."、"请..."形式的祈使/指令语调，而非口语化的描述方式。技能是指令书。

### 节约上下文

上下文窗口是公共资源。自问每个句子是否对得起 Token 成本:
- "Claude 已经知道的内容吗？" → 删除
- "没有这个说明 Claude 会犯错吗？" → 保留
- "一个具体示例比长篇说明更有效吗？" → 用示例替代

## 3. 输出格式定义模式

在产出物格式重要的技能中使用:

```markdown
## 报告结构
严格按照以下模板:

# [标题]
## 摘要
## 核心发现
## 建议
```

格式定义要简洁，包含实际示例效果更好。

## 4. 示例编写模式

示例比长篇说明更有效:

```markdown
## 提交信息格式

**示例 1:**
输入: 添加基于 JWT Token 的用户认证
输出: feat(auth): 实现 JWT 基础认证

**示例 2:**
输入: 修复登录页面密码显示按钮不工作的 Bug
输出: fix(login): 修复密码显示切换按钮功能
```

## 5. Progressive Disclosure 模式

### 模式 1: 按领域分离

```
bigquery-skill/
├── SKILL.md (概览 + 领域选择指南)
└── references/
    ├── finance.md (营收、计费指标)
    ├── sales.md (商机、流水线)
    └── product.md (API 使用量、功能)
```

用户询问营收时只加载 finance.md。

### 模式 2: 条件性详情

```markdown
# DOCX 处理

## 文档生成
使用 docx-js 创建新文档。→ 参考 [DOCX-JS.md](references/docx-js.md)。

## 文档编辑
简单编辑直接修改 XML。
**需要追踪修订时**: 参考 [REDLINING.md](references/redlining.md)
```

### 模式 3: 大型参考文档文件结构

300行以上的参考文档文件在顶部包含目录:

```markdown
# API 参考文档

## 目录
1. [认证](#认证)
2. [端点列表](#端点列表)
3. [错误代码](#错误代码)
4. [速率限制](#速率限制)

---

## 认证
...
```

## 6. 脚本打包判断标准

观察测试执行中智能体的转录记录。出现以下模式时为打包对象:

| 信号 | 措施 |
|------|------|
| 3个测试中3个都生成相同的辅助脚本 | 打包到 `scripts/` |
| 每次执行相同的 pip install/npm install | 在技能中明确依赖安装步骤 |
| 重复相同的多步骤方法 | 在技能正文中描述为标准流程 |
| 每次遇到类似错误后应用相同的规避方法 | 在技能中记录已知问题及解决方案 |

打包的脚本必须经过执行测试。

## 7. 数据 Schema 标准

为保持技能间数据交换的一致性，使用标准 Schema。可用于 Harness 中生成技能的测试/评估。

### eval_metadata.json

每个测试用例的元数据:

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "用户的任务提示词",
  "assertions": [
    "产出物中包含 X",
    "生成了 Y 格式的文件"
  ]
}
```

### grading.json

基于 assertion 的评分结果:

```json
{
  "expectations": [
    {
      "text": "产出物中包含 '北京'",
      "passed": true,
      "evidence": "第3步确认 '北京地区数据提取'"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  }
}
```

**字段名注意:** 严格使用 `text`, `passed`, `evidence` (禁止使用 `name`/`met`/`details` 等变体)。

### timing.json

执行时间/Token 测量:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

在子智能体完成通知中立即保存 `total_tokens` 和 `duration_ms`。这些数据仅在通知时刻可访问，之后不可恢复。

## 8. 技能中不应包含的内容

- README.md, CHANGELOG.md, INSTALLATION_GUIDE.md 等附加文档
- 技能创建过程的元信息 (测试结果，迭代历史)
- 面向用户的说明书 (技能是面向 AI 智能体的指令)
- Claude 已经知道的通用知识

## 9. 技能复用设计

创建新技能前，确认与已有技能的重叠。反复构建 Harness 容易导致功能重叠的技能以不同名称累积。

| 情况 | 措施 |
|------|------|
| 已有技能完全包含新功能 | 禁止新建 — 将已有技能连接到智能体 |
| 已有技能部分包含且可泛化 | 扩展已有技能进行泛化 |
| 部分包含但有意进行领域特化 | 新建 — 作为独立技能维护 |
| 功能范围完全不同 | 新建 |

**原则:** 一个技能专注于一个角色时复用性最高、重复最少。如果角色有两个以上，先考虑是否可以分离。

### 泛化到什么程度

泛化可以无限进行，因此应在**有意设定的责任范围**内停止。保留有意的领域特化，仅移除偶然的依赖。

示例: "fintech 风险评估 PDF" 技能

| 步骤 | 结果 |
|------|------|
| 移除 fintech 依赖 | "评估结果 PDF" — 如果责任范围是评估报告，则到此为止 |
| 移除评估依赖 | "PDF 格式化" — 如果已存在，则不要创建单独的技能，而是复用 |

如果责任范围是 "fintech 风险评估" 的有意特化，则不进行泛化，作为独立技能维护。

依赖该技能的智能体的行为可能发生变化。扩展前确认依赖关系，并在 description 中反映扩展后的使用范围。
